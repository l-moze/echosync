3天极速开发：开源 AI 同传工作站技术选型与实现报告 (国内友好版)在 3 天的开发周期和“个人开发者、开源免费、国内网络友好”的绝对约束下，系统架构必须采用“成熟的 WebRTC 编排框架 + 优秀的国产单点模型”。核心架构基调： Next.js 前端 + LiveKit 实时音视频网关 + Python Agent 后端推理编排 + 纯开源/白嫖模型链路。以下是针对该架构的详细技术选型、架构设计与落地优化指南。1. 核心技术栈与开源组件选型为了兼顾开发速度与极低成本，我们将繁重的底层音视频流处理委托给开源框架，将推理大脑替换为国内顶尖服务。实时音视频流转（核心中间件）：LiveKit简介：完全开源的 WebRTC 实时流媒体服务器，自带前后端 SDK。选型理由：在 3 天内手写 WebSocket 并处理音频重采样、丢包重传和静音检测（VAD）是不现实的。LiveKit 提供了现成的 React 组件和 Python livekit-agents 框架，内置 Silero VAD 与对话轮次检测。开发者可以利用其 Cloud 免费额度（每月 50GB）免部署直接起步。ASR 语音识别：阿里 FunASR简介：通义实验室开源的端到端语音识别模型，支持 50+ 语种，中文识别率国内顶尖。选型理由：支持实时低延迟流式转写（如 Fun-ASR-Nano 或 SenseVoice），不易出现幻觉，自带标点恢复功能。翻译大脑（LLM）：DeepSeek-V3简介：当前国内性价比极高、推理极强的大语言模型。选型理由：接口完美兼容 OpenAI 格式。可以直接复用 LiveKit 的 OpenAI 插件，只需修改 base_url 和 API Key，极大缩短开发对接时间。TTS 语音合成：edge-tts简介：一个利用微软 Edge 浏览器内置在线语音服务的 Python 开源模块。选型理由：完全免费、零配置、无需 API Key，可以直接集成到 Python 后端代码中生成自然流畅的多种语言语音。2. 架构设计与数据流转本方案采用经典的 Agent 架构，前后端职责彻底解耦。2.1 拓扑结构客户端 (Next.js)：使用 @livekit/components-react 接入麦克风，将原始音频推流到 LiveKit Room。LiveKit Server：作为中间件转发 WebRTC 音频流与数据通道（Data Channel）。Python Agent (后端)：基于 livekit-agents 运行的守护进程。入耳：拦截用户音频，送入 FunASR 进行实时转写。大脑：通过 LiveKit 框架将文字送给 DeepSeek-V3 翻译。出嘴：将翻译结果送入 edge-tts 转换为音频，推回给前端播放。2.2 定制化插件方案 (Custom Node)由于 FunASR 和 edge-tts 目前并非 LiveKit 官方内置的商业插件，开发者需要利用 LiveKit 提供的底层接口编写自定义组件：自定义 STT（语音识别）：继承 livekit.agents.stt.STT 基类。在子类中集成 FunASR 的 WebSocket 客户端 API（如使用 Funasr_websocket_recognizer 处理音频切片 chunk_size="5,10,5"），并将结果作为 SpeechEvent 返回给 LiveKit。自定义 TTS（语音合成）：继承 livekit.agents.tts.TTS 基类。在 synthesize 方法中调用 edge_tts.Communicate 获取音频，并将其返回为 ChunkedStream 交给 WebRTC 推流。3. 72 小时开发排期与代码级落地 (Task List)Day 1：基础设施、LiveKit 通道打通上午：注册 LiveKit Cloud，获取 LIVEKIT_URL、LIVEKIT_API_KEY、LIVEKIT_API_SECRET。使用 npx create-next-app@latest 初始化前端，安装 @livekit/components-react。下午：实现前端 LiveKit 房间连接逻辑，渲染默认的麦克风采集按钮。初始化 Python 环境：uv init --bare，接着 uv add "livekit-agents[silero,turn-detector]~=1.4" python-dotenv websocket-client。晚上：运行基础的 Python Agent，确保前端说话时，后端的 Python 进程能够成功接收到音频帧事件。Day 2：核心开源模型深度集成上午 (FunASR 接入)：利用 Docker 或直接在服务器运行 FunASR 实时服务：funasr-server --device cuda --port 10095。在 Python Agent 中编写自定义 STT 类。通过 WebSocket 将 LiveKit 拦截到的音频帧 feed_chunk 发送至 FunASR，获取流式中文/英文文本。下午 (DeepSeek 翻译引擎)：安装 OpenAI 插件库。在初始化 AgentSession 时，使用 DeepSeek API：llm = openai.LLM(model="deepseek-chat", base_url="https://api.deepseek.com/v1")。设置系统 Prompt，使其严格扮演“同声传译员”角色，不对文本进行额外聊天回复。晚上 (Edge-TTS 接入)：编写自定义 TTS 类继承 livekit.agents.tts.TTS。在类中调用 edge_tts.Communicate(text=text, voice="zh-CN-XiaoxiaoNeural") 获取音频字节流，封装入 ChunkedStream 供 LiveKit 消费。Day 3：字幕状态机、防闪烁与最终体验优化上午：打通数据流：通过 LiveKit 的数据通道 (Data Channel) 或 RPC，将 FunASR 的源语言字幕和 DeepSeek 的翻译字幕推送到前端客户端。下午：前端防闪烁渲染：使用 React 状态将字幕划分为“已确认（高对比度）”和“识别中/翻译中（浅色半透明）”。依赖于 FunASR 返回的最终/中间结果标识来驱动。断线重连与异常捕获：在 Python 端做好 WebSocket 断开重连机制，以及 edge-tts 请求失败时的容错处理。晚上：将前端静态页面部署至 Vercel 或 Netlify。将 Python Agent 后端部署在腾讯云/阿里云的轻量应用服务器上，使用 nohup 或 systemd 维持进程运行。完成产品交付。4. 架构级优化与成本控制建议VAD (静音检测) 前置：务必在 LiveKit 层启用内置的 Silero VAD 插件。只有当用户说话时，才将数据流发送到 FunASR 和 DeepSeek。这不仅降低了服务器带宽，还极大减少了 LLM 推理的 Token 浪费。TTS 缓存机制：对于频繁出现的简短词语（如“你好”、“谢谢”），可以在 Python 内存中维护一个 LRU 缓存。当翻译结果命中时，直接返回本地缓存的音频流，不再调用 edge-tts，降低延迟。极低成本优势：在此架构中，前端托管免费，STT (FunASR) 免费开源，TTS (edge-tts) 白嫖免费，唯一的变量成本仅为极便宜的 DeepSeek-V3 Token 消耗。这使得该系统非常适合个人开发者长期维护与试错。
