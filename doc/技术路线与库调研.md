# 当前代码问题与库调研

## 结论

当前 EchoSync 的最小主线已经从早期 LiveKit 方案切到更直接的本地桌面链路：

```text
Electron Windows 系统声音 / 麦克风
  -> Web Audio 转 16k mono PCM16
  -> FastAPI WebSocket `/v1/realtime/sessions/{session_id}`
  -> Python Agent ASR / 翻译 / 修正
  -> `/v1/caption/events`
  -> Electron 透明悬浮字幕窗
```

LiveKit 仍是后续可选传输层，不是当前已接通的主链路。这个判断来自代码状态：`LiveKitAgentBridge.run()` 仍然是 `NotImplementedError`，而 `caption_ws.py` 已经挂载了 realtime 输入和 caption 输出两个 WebSocket。

## 当前代码问题

| 问题 | 影响 | 建议 |
|---|---|---|
| 默认 `ECHOSYNC_ASR_PROVIDER=mock` | 真实 PCM16 音频进入 mock 时不会得到有效转写，容易误以为端到端失败 | 文档和 UI 都要明确：`mock` 只用于事件演示；真实识别使用 `funasr` 或 `voxtral` |
| 混音、文件源 UI 先于实现 | 用户看到入口，但实际 renderer 当前主要支持系统声和麦克风 | 混音和文件未实现前标记为不可用 |
| `ScriptProcessorNode` 仍在采集链路中 | 可跑 MVP，但长期运行和低抖动不稳 | 下个迭代迁移到 `AudioWorklet`，不要继续扩展旧 API |
| Agent 不会随 Desktop 自动启动 | 用户直接点桌面端时，8766 未启动会连接失败 | 先做启动前健康检查；之后再考虑 Desktop 拉起 Python sidecar |
| 8765 与 8766 两个服务容易混淆 | 8765 是纯 ASR，8766 是完整同传链路 | 文档统一：桌面真实链路用 8766；8765 只做 ASR 单点调试 |
| LiveKit 文档仍像当前主线 | 其他 Agent 容易沿错方向开发 | 明确 LiveKit 是后续候选，不阻塞当前本地 MVP |
| 依赖里有未使用预留项 | 环境变重，安装慢，排查困难 | 把 LiveKit、edge-tts、OpenAI、media 等按 optional extra 或后续依赖管理 |

## 库调研与选择

### Windows 系统声音采集

当前用 Electron `desktopCapturer` / `getDisplayMedia` 是合理的 MVP 路线。Electron 官方示例已经支持通过 `setDisplayMediaRequestHandler` 授权屏幕源并设置 `audio: "loopback"`，适合快速验证 Windows 系统声音链路。

生产级替代路线是 WASAPI loopback。Microsoft 文档明确说明 WASAPI loopback 可以捕获渲染端点正在播放的音频流，但这意味着要引入 native addon、Rust/C++ sidecar 或 Python sidecar。MVP 阶段不应该先造这个轮子。

### 麦克风采集

麦克风应该直接使用 `navigator.mediaDevices.getUserMedia({ audio: true })`。MDN 明确该 API 用于访问用户音视频输入，并且需要用户授权。当前代码把麦克风也走 `getDisplayMedia`，这是需要修正的实现断点。

### Web Audio 处理

`ScriptProcessorNode` 已被 MDN 标记为 deprecated，并建议迁移到 `AudioWorkletNode`。因此当前 `ScriptProcessorNode` 只保留为 MVP 快速验证，不要在它上面继续堆复杂缓冲、VAD 或重采样策略。后续应把 PCM 转换移到 `AudioWorklet`，主线程只负责 WebSocket 发送和 UI。

当前 renderer 已增加轻量音频门控：

```text
静音低于阈值：不发送音频正文 frame
响度超过阈值：开始发送约 80ms 的 `pcm16.binary.v1` PCM binary frame
连续静音约 640ms：把上一块活跃音频标记为 is_final=true
停止采集：flush pending chunk，再发送 audio.end
```

注意：`audio-gate.ts` 的默认 `chunkMs=240` 是通用兜底值；Desktop 真实实时链路在 `realtime-audio-client.ts` 中显式传入 `AUDIO_FRAME_DURATION_MS=80`。旧的 JSON `audio.chunk` / `pcm_base64` 只作为兼容路径和部分测试协议保留，不是当前 Desktop 主链路。`audio.start` 会携带 `asr_provider` 和 `asr_latency_mode`，Agent 可以按会话覆盖 `.env` 默认 ASR provider。

这不是最终 VAD，只是避免静音和背景空流持续打满 ASR。生产级仍建议迁移到 `AudioWorklet` 后接入 Silero VAD、WebRTC VAD 或 WASAPI 层活动检测。

### 原始音频录制与回放

当前会话回放记录使用浏览器标准 `MediaRecorder` 保存原始 `MediaStream`，与 ASR 发送链路分离。这样 ASR 可以继续做门控、重采样和低延迟二进制帧发送，而复盘页可以保留完整音频、静音和停顿，用于按时间帧回放。

后续如果需要波形、拖选片段、循环播放或局部重转写，可接入 `wavesurfer.js` 和 Regions 插件。MVP 暂不引入它，避免在当前“录音 + 片段级高亮”还未落盘前增加依赖和 UI 复杂度。

### FunASR

FunASR 官方流式示例使用 `AutoModel(model="paraformer-zh-streaming")`，逐块调用 `model.generate()`，并要求跨 chunk 维护 `cache`、最后一块设置 `is_final=True`。当前 `FunAsrTranscriber` 的方向与官方模式一致：模型和 cache 封装在 `Transcriber` 适配器内部，管道不依赖 FunASR SDK。适配器遇到上游 endpoint final 会 flush 当前窗口并重置 cache，避免一句话的流式状态污染下一句话。

当前 Desktop 传输帧是 80ms，但 FunASR 推理不应跟着 80ms 一帧一调用。`FunAsrTranscriber` 内部按会话延迟模式聚合 PCM：`low_latency` 约 320ms，`balanced` 使用 `FUNASR_CHUNK_MS` 默认 600ms，`accuracy` 使用约 900ms；收到 final 或迭代结束时 flush 剩余音频。这样保留低延迟传输和准确 telemetry，同时避免本地模型调用过碎导致吞吐下降、准确率变差。运行时用 `funasr_inference_chunk.transport_frames` 和 `input_audio_ms` 验证聚合是否生效，而不是凭主观体感判断优化。

`services/asr/semantic_chunker.py` 已落地第一版：`SemanticAudioChunker` 支持 soft cut、hard cut、overlap，适合未来 batch ASR 或非流式模型；`SemanticEndpointTracker` 服务 FunASR 这种流式 ASR，只做 endpoint 标记，不缓存音频。它已经支持可插拔 `FrameVadDetector`，`LiveKitSileroFrameVadDetector` 可把 `livekit.plugins.silero` 异步事件流转换成同步人声状态；但本地 30 秒英文视频样本里，开启 Silero 没有增加 committed 片段或提前 soft endpoint，平均 ASR 耗时从 93ms 升到 111.6ms，并出现 10 条 `inference is slower than realtime` 警告。因此 MVP 默认关闭 `FUNASR_VAD_ENABLED`，继续依赖 hard timeout 兜底；Silero VAD 保留为后续调参实验开关。

### Voxtral Realtime

Mistral 官方文档把 Voxtral Realtime 定位为 live transcription，支持 `voxtral-mini-transcribe-realtime-2602`，并提供 `target_streaming_delay_ms` 平衡延迟与准确率。它适合作为英语技术分享、国际会议、网课的云端 ASR 备选，但不要替代翻译器、字幕状态机或修正引擎。

### Deepgram / Azure / OpenAI Realtime 候选

后续低延迟/准确率评测应优先接 Deepgram 或 Azure 这类成熟 streaming ASR：它们能表达 interim/final、endpointing、稳定度或词级置信度，比较适合 EchoSync 的可修正字幕状态机。当前代码尚未实现 Deepgram/Azure provider，后端也不会接受这些名字作为 `audio.start.asr_provider`。OpenAI Realtime 更适合作为端到端语音/同传引擎候选，不建议先强行塞进 ASR-only provider。

### LiveKit

LiveKit Agents 是成熟的语音 Agent 框架，能组合 STT、VAD、LLM、TTS、turn detection 和房间音频 I/O。它适合多人房间、浏览器远程音轨、云端会议场景。当前 Windows 本地悬浮字幕 MVP 不引入 LiveKit 房间、token、信令和 Agent worker；`livekit.plugins.silero` 只作为 FunASR endpoint detector 的可选实验能力，默认不进入热路径。

### DeepSeek / OpenAI-compatible 翻译

DeepSeek API 兼容 OpenAI 风格的 chat completion，并支持流式输出。当前 `DeepSeekTranslator.stream_translate()` 的路线是合适的：只在翻译适配器里处理 SDK 和 prompt，管道只消费 `Translator` / `StreamingTranslator` 抽象。默认调度策略只把 `stable/committed` 文本送入 DeepSeek，`partial` 只用于源文实时显示；如果为了极低延迟开启 partial 预翻译，必须作为显式实验开关，并通过日志统计额外请求量。

### DeepL / 专用机器翻译 API

DeepL 官方文本翻译 API 是 `/v2/translate` 请求/响应式接口，单次请求可传最多 50 个 `text`，并支持 `context`、术语表、模型类型偏好等参数；它不适合当作 token 级实时流式翻译器。EchoSync 应把这类供应商实现为 batch-only `Translator.translate()`：实时直播场景里只在稳定 checkpoint 后补译文，录播视频或可直接访问完整音轨时可以缓冲 1-3 个片段打时间差，以换取更稳定的译文质量。

这个边界能避免重复造轮子：DeepL 负责高质量文本翻译，EchoSync 负责 ASR 时间戳、字幕时间差、术语命中和前端修订显示。不要为了统一接口去伪造 DeepL 的 `stream_translate()`。

### FastAPI WebSocket

FastAPI 官方支持 WebSocket 路由、收发 JSON 和连接生命周期管理。当前 `caption_ws.py` / `realtime_ws.py` 用它承载本地音频输入和字幕事件输出，符合“先接通本地 MVP，后续替换传输层”的策略。

## 依赖治理建议

| 依赖 | 当前状态 | 建议 |
|---|---|---|
| `fastapi`、`uvicorn`、`python-dotenv` | 已实际使用 | 保留默认依赖 |
| `funasr`、`modelscope` | 延迟 import，真实本地 ASR 才需要 | 保持 optional extra `funasr` |
| `mistralai[realtime]` | Voxtral ASR 才需要 | 保持 optional extra `voxtral` |
| `livekit-agents[silero,turn-detector]` | 已用于 FunASR Silero VAD adapter；默认关闭，未使用房间/Agent worker | 短期保留依赖方便继续调参；若 VAD 长期不是默认路径，应拆到 optional extra |
| `edge-tts` | 有实现但未进入主链路 | 移到 optional extra `tts` |
| `openai` | DeepSeek 翻译才需要 | 可移到 optional extra `deepseek` |
| `imageio-ffmpeg` | 媒体文件 demo 才需要 | 可移到 optional extra `media` |
| `websocket-client` | 当前源码未使用 | 移除或需要时再加 |
| `@livekit/components-react`、`livekit-client` | Web 端未使用 | 后续 LiveKit UI 接入时再加 |
| `zod`、`cross-env` | 当前未使用 | 移除或等真实使用再加 |

## 下一步优先级

1. 修正 Desktop 音频源分支：`windows-system` 用 `getDisplayMedia`，`microphone` 用 `getUserMedia`，未实现源禁用。
2. 给 Desktop 开始同传前加 Agent 8766 健康检查，避免用户误以为 UI 坏了。
3. 把 renderer 音频处理从 `ScriptProcessorNode` 迁移到 `AudioWorklet`。
4. 继续用真实视频链路采集 FunASR no-VAD / VAD / 其他 ASR provider 的 ASR RTF、committed 间隔和字幕端到端延迟；当前证据支持 MVP 默认 no-VAD hard timeout。
5. 依赖瘦身：把未接线或可选的大依赖移到 optional extra，降低新环境安装成本。

2026-06-06 FunASR 真实 30 秒样本补测：

| 模式 | 输出行 | committed | 平均 ASR 耗时 | 平均 ASR RTF | 最终 e2e RTF | Silero 慢实时警告 |
|---|---:|---:|---:|---:|---:|---:|
| `FUNASR_VAD_ENABLED=false` | 38 | 9 | 93.0ms | 0.30 | 0.86 | 0 |
| `FUNASR_VAD_ENABLED=true` | 38 | 9 | 111.6ms | 0.36 | 0.79 | 10 |

结论：Silero VAD 当前没有在该样本上兑现更早切句收益，反而增加本地推理开销；默认关闭是更稳的 MVP 策略。
