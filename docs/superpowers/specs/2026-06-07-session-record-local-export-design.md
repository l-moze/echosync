# Session Record Local Export Design

## Context

The current record detail UI has export buttons for Markdown, TXT, and SRT, but the action copies generated text to the clipboard instead of saving a file. Users expect the blue export action in the review detail screen to publish the reviewed bilingual transcript to a local file.

The desktop app already stores complete session records in the Electron main process and exposes record export text through `window.echosyncDesktop.sessionRecords.export(id, format)`. This design extends that path to save files through Electron's main process, keeping filesystem access out of the renderer.

## Goals

- Export a selected session record to a user-chosen local path.
- Support real `.docx` plus `.md`, `.txt`, `.srt`, `.json`, and `.csv`.
- Keep existing copy-to-clipboard behavior available as a fallback action.
- Use the same full session record data shown in the review detail page, including edited source/target text when present.
- Show clear renderer status for saved, canceled, and failed exports.

## Non-Goals

- Exporting audio files is out of scope for this change.
- Batch exporting multiple records is out of scope.
- Cloud publishing, share links, and automatic opening after export are out of scope.
- Rich Word styling beyond a clean readable document is out of scope.

## Formats

`docx` is a real Office Open XML document generated by the `docx` npm package. It contains the record title, metadata, summary, and bilingual transcript sections.

`markdown` uses the existing Markdown structure and saves with `.md`.

`txt` is plain UTF-8 text with bilingual segment pairs.

`srt` keeps original segment timestamps and includes source and target text.

`json` exports a stable, human-readable JSON object for downstream processing.

`csv` exports one row per segment with columns for index, start time, end time, source text, target text, revision state, and patch count.

## Architecture

Shared record serialization lives in `apps/desktop/src/shared/session-records.ts`. Text formats return strings. `docx` creation lives in a focused main-process module because it produces binary data and depends on a Node package.

The renderer asks the preload API to save the selected record with a format. The preload bridges to a main-process IPC handler. The main handler loads the record, creates the export payload, shows `dialog.showSaveDialog`, writes the selected file with `fs.writeFile`, and returns `{ canceled, path }`.

The existing `sessionRecords.export(id, format)` API remains available for copy behavior and tests. A new `sessionRecords.saveExport(id, format)` API performs local file export.

## UI Behavior

The record detail actions show explicit format buttons: `DOCX`, `Markdown`, `TXT`, `SRT`, `JSON`, and `CSV`. The primary export path saves to disk. A small secondary action keeps Markdown copy behavior.

Status messages:

- Saved: `<FORMAT> 已导出到本地`
- Canceled: `已取消导出`
- Failed: `导出失败`
- Copied: `Markdown 已复制`

The save dialog default filename is based on the record title and format extension. Invalid filename characters are sanitized in the main process.

## Error Handling

If the record is missing, the main process throws `未找到会议记录：<id>` and the renderer shows export failure.

If the user cancels the save dialog, no file is written and the renderer shows cancellation.

If writing fails, the error is logged in the renderer and the status becomes `导出失败`.

## Testing

Shared tests cover `json` and `csv` serialization and format metadata.

Main-process tests cover export payload generation and safe default file names without invoking a real OS dialog.

Renderer contract tests verify that the detail page exposes all six formats, calls `saveExport`, and preserves Markdown copy.

Typecheck and desktop tests are the verification gates.
